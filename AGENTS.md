# ACGTI AGENT 工作规范

本文件适用于仓库根目录及其所有子目录。除非用户明确给出更高优先级指令，否则在本仓库内工作的 Agent 必须遵守以下规则。

## 1. 项目定义

### 1.1. ACGTI 是什么

`ACGTI` = `ACG Type Indicator`

它是一个面向二次元语境的娱乐型角色原型测试，不是严肃心理测量工具，也不应被实现为医学、心理咨询或人格诊断产品。

它测的不是“你真实的人格类型”，而是：

- 你在二次元叙事里更像什么类型的人
- 你在作品里通常会处于什么角色位
- 你的气质、社交方式、情绪表达、行动风格，更接近哪种“人设原型”

### 1.2. 产品定位

本项目默认按以下产品定位推进：

- `ACGTI` 不是 `MBTI` 套壳，也不是现实人格鉴定工具。
- `ACGTI` 的核心输出是“角色原型”与“叙事位置”，而不是生硬的四字母标签。
- `ACGTI` 首先是一个重内容表达、视觉体验和分享感的静态网站项目。
- `ACGTI` 的价值主要来自题目设计、结果文案、世界观包装、页面氛围与交互体验，而不是复杂后端。

## 2. 默认工作模式

收到用户任务后，必须按以下顺序推进，不允许跳过“先理解再实现”的过程：

1. 接收用户指令
2. 阅读现有代码、文档、素材和配置
3. 形成实现方案
4. 执行实现
5. 自我验证
6. 交付成果

执行要求：

- 必须先理解仓库现状，再进行修改；即使仓库刚初始化，也要先确认目录结构与已有文件。
- 必须尽量形成完整闭环，不要只停留在建议层面。
- 应优先自主完成可落地工作，仅在缺少关键输入且无法合理推断时再询问用户。
- 完成后必须进行最基本的自我验证；若因为环境或项目阶段限制无法验证，必须明确说明。

## 3. 内容与产品表达要求

- 所有用户可见文案应优先使用“角色原型”“人设气质”“剧情位置”“叙事功能”等表达。
- 避免把结果写成严肃人格结论，禁止暗示专业心理评估、医学判断或现实人格定型。
- 题目、选项、结果文案、页面标题都应保持 ACG 语境一致性，避免普通互联网测试站口吻。
- 结果内容要更像“设定档案”或“角色卡”，而不是简单打标签。
- 若需要使用类型编码，编码只能作为内部结构辅助；对外展示应优先展示完整原型名称和描述。

## 4. 技术实现边界

- 默认将本项目视为静态网站或前端优先项目处理。
- 若无明确需求，不主动引入后端服务、用户系统、数据库、鉴权、消息队列或云函数。
- 若需保存测试结果，优先考虑本地存储、纯前端状态或静态可部署方案。
- 任何技术选型都应服务于页面体验、内容组织和可维护性，而不是炫技。
- 新增依赖前必须考虑是否真的必要，避免在项目早期引入过重工具链。

## 5. 视觉与交互要求

- 页面风格应体现明确的二次元叙事气质，避免通用 SaaS 风格或模板化落地页。
- 视觉设计应优先强调氛围、角色感、章节感和结果揭示过程。
- 结果页应具备足够的“展示价值”，适合截图、分享或作为角色卡呈现。
- 如无现成设计系统约束，可适当采用更鲜明的字体、配色、分区和动效，但必须保持可读性。
- 移动端体验不可被忽略，题目作答、结果切换和分享展示都应兼顾手机端。

## 6. 代码与文件组织要求

- 保持良好的模块拆分，避免把页面逻辑、题库数据、结果映射和样式全部堆在一个文件里。
- 题目数据、类型定义、结果描述、视觉资源引用应尽量分层组织。
- 在复杂映射逻辑、评分逻辑、结果归类逻辑处补充简洁注释，说明设计意图。
- 优先写清晰、可维护的实现，不要为了“智能感”牺牲可读性。

## 7. Git 与提交要求

- 每完成一个功能，就应进行一次本地 Git 提交。
- 提交前必须先检查工作区，只提交与当前任务直接相关的文件。
- 提交信息必须使用中文，并采用“标题 + 空行 + 正文”的形式。
- 提交前应用 `git status` 和 `git diff --staged` 复核内容。
- 绝对禁止执行 `git push`。

推荐格式：

```text
<type>(<scope>): <中文摘要>

背景：
- 说明为什么要做这次改动。

实现：
- 说明改了什么、采用了什么策略。

验证：
- 说明做了哪些检查、构建或人工验证。

影响：
- 说明影响范围、限制和后续事项。
```

## 8. 安全与边界要求

- 绝不在仓库中硬编码 API Key、令牌、密码或其他敏感信息。
- 不要把 `ACGTI` 描述成心理诊断、医学评估或可替代专业意见的工具。
- 若用户提出会显著偏离本项目娱乐定位的设计，应先指出风险，再执行用户明确确认后的方案。

## 9. 实施原则

- 先读后改：先理解当前实现和项目方向，再动手。
- 内容优先：优先保证世界观、题目设计、结果表达和页面体验的一致性。
- 轻量优先：没有充分理由时，不把简单问题升级成复杂架构。
- 结果导向：交付时明确说明做了什么、怎么验证、还有什么限制。
- 尊重现状：不得随意覆盖或回退用户已有改动。
